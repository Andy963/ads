## 2024-05-24 - [Path Traversal bypass via `startsWith` validation]
**Vulnerability:** A path traversal vulnerability existed in the static file serving logic (`server/web/server/httpServer.ts`). The code validated requested file paths against the base directory (`distClientDir`) using `!resolved.startsWith(distClientDir)`. This allowed partial directory matching, meaning a path like `/app/dist/client-secrets.txt` (requested via `%2E%2E%2Fclient-secrets.txt` or `..%2fclient-secrets.txt` normalized to `/client-secrets.txt` relative to base dir) would bypass the security check because it literally *starts with* `/app/dist/client`.
**Learning:** Using `String.prototype.startsWith()` directly on a directory path without appending the trailing path separator (`path.sep`) creates a known path traversal bypass vector through partial directory name matches.
**Prevention:** To validate paths against a base directory, always ensure the base directory string ends with a path separator (`path.sep`) before calling `startsWith()`, like `resolved.startsWith(baseDir + path.sep)`, and also handle the exact match case `resolved === baseDir` if the base directory itself is a valid requested path.
