## Introduction
- 问题背景：Web 端对话流在执行/流式生成时缺少“中断/停止”入口，无法及时终止长耗时或误触发的命令（如删除文件）。
- 根因摘要：Web UI 未暴露中断控制，仅 Telegram 有 `/esc`；后端已有基于 AbortSignal 的软中断能力可复用。
- 期望成果：提供输入框右侧的停止按钮，在有活跃请求时可一键中断当前流，终止后端执行并提示用户输出可能不完整，空闲时不可用。

## Scope
- In Scope：
  - Web 前端新增停止按钮（方块图标）放置在输入框右侧，行为与 Telegram `/esc` 一致。
  - 前端状态管理：检测当前是否有活跃请求；执行时启用按钮，空闲时禁用。
  - 后端接口：调用现有中断能力（不杀进程），确保当前活跃流（LLM+命令串行）收到 abort 信号。
  - UI 反馈：点击后立即显示“已中断/输出可能不完整”提示，并允许继续发送新消息。
- Out of Scope：
  - 杀掉后台服务进程或跨用户/跨平台批量中断。
  - 新增并行多流管理（当前假设单活跃流）。
  - Telegram 行为调整（保持 `/esc` 现状）。

## Functional Requirements

### Requirement 1: Web 停止按钮中断当前活跃流
- 概述：在 Web 输入框右侧新增方块停止按钮。仅在有活跃请求时可点击；点击后向后端发 abort 信号，当前流（LLM/命令串行）应尽快停止，前端提示中断状态，随后可继续发送消息。

**User Story:** 作为 Web 用户，我希望正在执行的回复或命令可以随时停止，以便在误操作或长耗时场景下快速终止并继续其他操作。

#### Acceptance Criteria
- [ ] WHEN Web 前端检测到有活跃请求（流式输出中）， THEN 停止按钮可点击；空闲时按钮禁用或灰显（验证：手动/单测状态机）。
- [ ] WHEN 用户点击停止按钮， THEN 立即向后端发送中断指令，当前流终止，不再追加新 token/命令输出（验证：前后端联调，查看流停止）。
- [ ] WHEN 中断发生， THEN 前端在当前对话区域提示“已中断，输出可能不完整”，且输入框可继续发送新消息（验证：UI 手动）。
- [ ] IF 中断时后端已部分执行命令， THEN 前端不应假定回滚，提示用户可能需要检查影响（验证：文案/日志）。
- [ ] WHEN 再次发起新请求， THEN 不受上次中断状态影响，停止按钮按新请求状态启用/禁用（验证：手动）。

#### Validation Notes
- 日志 / 监控：记录停止事件（时间、会话标识、是否成功触发 abort），可落在前端控制台或后端日志。
- 手动验证：启动 Web，触发长回复，点击停止，确认流停止、提示出现、可继续发送；重复多次验证状态切换。

## Non-Functional Requirements
| Category | Requirement | Metric / Threshold | Validation |
| -------- | ----------- | ------------------ | ---------- |
| 可靠性 | 停止指令发出后应在可感知时间内生效 | 触发后 2s 内停止追加输出 | 手动观察/日志时间戳 |
| UX | 按钮状态与请求状态一致 | 无执行时禁用；执行时高亮可点 | 手动检查 |

## Observability & Alerting
- 日志：前端可选地记录停止操作；后端记录 abort 触发与目标会话/线程 ID。
- 指标 / Dashboard：不强制新增。
- 告警：不新增告警。

## Compliance & Security
- 权限：仅当前会话用户可触发自身请求的中断；不跨用户。
- 数据保护：无额外敏感数据存储。

## Release & Timeline
- 关键里程碑：需求 → 设计 → 实施 → 验证。
- 验收窗口：与设计/实现评审一致。
- 回滚方案：前端回退按钮入口；后端保留现有软中断逻辑，回退仅移除入口。
