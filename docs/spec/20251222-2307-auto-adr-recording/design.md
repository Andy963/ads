# Auto ADR Recording - 设计（A1：显式块触发 + 一决策一文件）

> 更新（2026-02-12）：ADS 不再支持用户侧 CLI 入口；本文档中关于 CLI 的内容仅作为历史背景描述，当前支持入口为 Web Console + Telegram Bot。

## 1. 文档概览
- 版本：v1
- 状态：Draft（待你确认后再 `/ads.commit design`）
- 作者：Codex（按 ADS 规则执行）
- 关联需求：`docs/spec/20251222-2307-auto-adr-recording/01-req.md`（req_x121o1be）
- 目标落地范围：ADS 核心代码（CLI/Web/Telegram 通过 agent hub 的对话输出）

## 2. 问题与目标

### 2.1 背景
ADR 决策目前大量散落在聊天记录与设计讨论中，缺少结构化沉淀与单点事实来源，导致重复讨论和追溯困难。

### 2.2 目标（对应需求）
- 仅当 agent 回复中显式包含 `<<<adr ... >>>` 控制块时，才触发记录（避免噪音）。
- 每条决策生成一份独立 ADR 文件：`docs/adr/000x-<slug>.md`。
- 自动维护 `docs/adr/README.md` 索引（按编号升序）。
- 对用户输出只追加“已记录 ADR”的简短提示，不污染主要回答。

### 2.3 非目标（本期不做）
- 不做“猜测式”从任意对话自动生成 ADR。
- 不在 `/ads.commit design` 时从设计稿抽取 ADR。
- 不与 `ads.db` 绑定存储 ADR 元数据（本期只写文件）。
- 不支持更新已有 ADR / Superseded 链接自动维护。

## 3. 交互与用户可见行为（FR-5）

### 3.1 控制块写法（对 agent 的约束）
触发格式固定为：

```text
<<<adr
{JSON}
>>>
```

JSON 至少包含：
- `title`: string（可选；若缺省则由系统生成）

建议字段：
- `status`: `Proposed | Accepted | Rejected | Deprecated | Superseded`
- `date`: `YYYY-MM-DD`（缺省则用生成当日）
- `context` / `decision` / `consequences` / `alternatives` / `references`
- 或直接提供 `body`（markdown）

### 3.1.1 标题缺省时的默认生成
当 `title` 缺省时，系统按以下优先级生成默认标题（避免要求用户必须提供标题）：
1. 若提供 `body`：取 `body` 中第一个 Markdown 标题（`# ...`），否则取第一行非空文本（截断到 60 字符）。
2. 否则若提供 `decision`：取 `decision` 第一句/前 60 字符。
3. 否则：使用 `Untitled ADR`。

### 3.2 对用户输出的呈现策略（不污染主回答）
- `<<<adr ... >>>` 视为**控制块**，不会原样展示给用户。
- 系统在发送最终回答前解析控制块并落盘；随后从输出中移除控制块。
- 成功时在回答末尾追加一段短提示（建议用分隔线降低干扰）：
  - `---`
  - `ADR recorded: docs/adr/0007-some-slug.md`
- 失败时同样仅追加一行提示失败原因，不影响主要回答内容输出。

### 3.3 “显式块触发”如何保证不漏（防遗忘策略）
你提到的风险是成立的：**如果 agent 忘记输出 `<<<adr ... >>>`，那本期方案就不会记录**。A1 的核心取舍是“低噪音、强可控”，代价是需要 agent 按约定输出控制块。

为降低“忘记写块导致永远不触发”的概率，本期在**不引入猜测式抽取（B 方案）**的前提下，采用以下组合策略：

1) **强约束的提示词契约（推荐，默认）**
- 在 ADS 主管 agent 的系统/开发者提示词中加入硬规则：
  - “当你做出架构/接口/数据模型等决策，并且该决策可能影响后续实现时，必须输出一个 `<<<adr ... >>>` 块。”
  - “输出前做一次自检：若本次回答包含‘我们决定/采用/将会/不做/选择方案’等决策性结论，且没有 ADR 块，则补上 ADR 块。”

2) **用户可控的补录机制（保证可追回）**
- 当用户说“记录 ADR: <标题>”或“把上面的决策写成 ADR”，agent 下一条回复只需要补一个 `<<<adr ... >>>` 控制块即可触发落盘（不要求自动回溯抽取全文）。
- 这让“当场忘记”不会变成“永远没法写”，而是变成“补一条控制块即可修复”。

3) **可选：轻量一致性校验（后续增强，非强制）**
- 若 agent 文本中出现了明确的 ADR 意图标记（例如包含“这是一个 ADR/ADR:”等），但没有输出 `<<<adr ... >>>`，系统可以在日志中记录一次 warning，便于发现“写了意图但漏了控制块”的情况。
- 注意：这里仅做提醒/监测，不做自动生成 ADR，以保持 A1 的显式触发边界。

## 4. 核心流程设计（FR-1 ~ FR-4）

### 4.1 流程总览
1. 在“最终回复输出”阶段对完整文本进行扫描，提取一个或多个 ADR 控制块（FR-1）。
2. 对每个控制块进行 JSON 解析：
   - 合法 JSON → 转为内部 `AdrRecord` 结构。
   - 非法 JSON → 忽略该块并记录用户可见警告（FR-1）。
3. 对每条 `AdrRecord`：
   - 选择下一个可用编号，构造文件名，确保不覆盖（FR-2）。
   - 渲染 Markdown 并写入 `docs/adr/`（FR-3）。
4. 成功写入后更新/生成 `docs/adr/README.md`（FR-4）。
5. 将“写入结果摘要”附加到最终回答尾部（FR-5）。

### 4.2 ADR 记录内部数据结构（概念）
最小字段：
- `title?: string`（可缺省，由系统生成）

可选字段：
- `status?: string`
- `date?: string`
- `body?: string`（markdown）
- `context?` / `decision?` / `consequences?` / `alternatives?` / `references?`

### 4.3 渲染规则（FR-3）
统一头部（总是生成）：
- 一级标题：`# ADR-0007: <title>`
- 元信息行（列表形式）：Status、Date

正文渲染策略：
- 若提供 `body`：直接原样追加在头部之后（仅补齐编号/元信息头）。
- 若未提供 `body`：按字段渲染标准章节（只渲染有内容的章节）：
  - `## Context`
  - `## Decision`
  - `## Consequences`
  - `## Alternatives`
  - `## References`（如为数组则逐条列出）

## 5. 文件命名与编号策略（FR-2）

### 5.1 目录与路径边界
- 仅允许写入 workspace 内 `docs/adr/`。
- 若 `docs/adr/` 不存在则创建目录（仅创建该目录，不做其他路径写入）。

### 5.2 编号选择（顺序、不覆盖）
- 读取 `docs/adr/` 下符合 `^\\d{4}-` 前缀的文件名，取最大编号 `maxN`，下一个候选为 `maxN + 1`。
- 为避免并发/重复导致覆盖：若候选文件已存在，则编号自增直到找到空闲编号。

### 5.3 slug 生成
- 由 `title`（若缺省则使用“默认生成标题”）生成 slug：小写、空白转 `-`、移除非字母数字与 `-`，并截断到合理长度（例如 60）。
- 若 slug 为空则使用 `untitled`。

## 6. 索引 README 生成（FR-4）

### 6.1 README 内容策略
为减少对人工编辑的破坏，采用“标记区块自动生成”策略：

- `docs/adr/README.md` 允许包含人工说明；自动生成的索引置于以下标记之间：
  - `<!-- ADS:ADR_INDEX_START -->`
  - `<!-- ADS:ADR_INDEX_END -->`

索引内容格式建议：
- 列表或表格均可，要求包含：编号、标题、文件链接（相对路径）。
- 按编号升序。

### 6.2 标题提取
- 优先从 ADR 文件的一级标题解析标题（例如 `# ADR-0007: xxx`）。
- 解析失败时回退到文件名 slug。

## 7. 配置与开关（NFR-3）
- 默认开启，不提供开关。

## 8. 错误处理与可观测性（FR-1 / FR-5）
- JSON 解析失败：忽略该块；在回答末尾追加一句可读警告（不中断对话）。
- 写文件失败（权限/路径/IO）：
  - 不影响主回答；在末尾追加失败原因（尽量可操作：例如路径不可写）。
- 索引更新失败：同上，主回答不受影响。

## 9. 安全与边界（NFR-1）
- 路径必须使用安全 join，并拒绝任何路径穿越。
- 不允许写入 `docs/adr/` 以外的路径。
- 不得删除或覆盖任何 `.db/.sqlite/.sqlite3` 文件（设计上不触碰数据库）。

## 10. 测试与验证策略（NFR-2 + 验收）
- 单元测试覆盖：
  - 多 ADR 块解析
  - 非法 JSON 容错（不抛出、不阻断）
  - 编号递增与不覆盖（包含“目标文件已存在”递增场景）
  - README 索引生成的排序与标题提取
- 手动验证：在 CLI/Web/Telegram 任一入口输出包含 ADR 控制块的回复，观察：落盘 + 索引更新 + 回复追加提示。

## 11. 方案对比与决策记录
- A（本期采用）：显式 `<<<adr ... >>>` 块触发，噪音低、可控。
- B（不做）：从任意对话“猜测”决策生成 ADR，误报风险高。
- C（不做）：在 `/ads.commit design` 抽取 ADR，耦合流程且不覆盖日常对话决策。

## 12. 已知风险与权衡
- 风险：agent 忘记输出 `<<<adr ... >>>` 会导致该次决策不被记录。
- 取舍：这是 A1 为了“低噪音/强控制”主动接受的代价。
- 缓解：通过“提示词硬约束 + 用户补录机制 +（可选）一致性校验日志”降低漏记影响面。
