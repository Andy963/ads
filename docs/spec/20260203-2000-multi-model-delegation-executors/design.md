# 多模型指派（一次性子执行器）- 设计

## 1. 设计概览
本设计的核心是把“使用哪个模型”从“主会话的全局状态”改为“指派任务的参数”。

用更容易理解的类比：
- 主会话 = 长期在线的“主管”，始终稳定，不被重置。
- 子执行器 = 临时叫来的“临时工”，每次带着一张任务卡，干完就走，不留下对话历史。

## 2. 为什么默认不用池化（上下文串扰问题）
如果把子执行器做成“池”，并复用其对话历史，就会出现：
- 上一个子任务的假设/路径/结论被带到下一个子任务，导致输出偏题或误改。
- 更严重的情况是把不相关的敏感信息带入后续任务（未来如果扩展到多用户/多项目更危险）。

因此本设计的默认策略是：**子执行器用完即丢**。未来若要做池化，只能池化“连接能力”，不能复用“对话历史”。

## 3. 高层流程（Data Flow）
### 3.1 指派触发
主会话根据任务复杂度/紧急程度选择是否指派：
- 简单、重复、偏机械：倾向使用更快模型（例如 `model-fast`）。
- 需要稳定推理、长上下文整合：留给主会话（例如 `model-primary`）。

### 3.2 任务卡（Task Spec）
每次指派必须携带结构化“任务卡”，其中包含：
- 目标（goal）
- 约束（constraints，例如禁止写文件、只允许产出 patch 等）
- 交付物（deliverables）
- 验收方式（verification commands / checks）
- **指定模型（model）**

示例（概念）：
```json
{
  "taskId": "t_example",
  "revision": 1,
  "goal": "Review the change and report risks",
  "model": "model-fast",
  "constraints": ["No file writes", "Return only a JSON result"],
  "deliverables": ["Review notes", "Potential regressions"],
  "acceptanceCriteria": ["Notes are actionable and specific"],
  "verification": { "commands": [] }
}
```

### 3.3 子执行器生命周期
每次指派：
1) 创建子执行器（使用同一套连接配置，仅 `model` 不同）
2) 子执行器执行任务（可多轮：读文件、分析、必要时调用工具）
3) 输出结构化结果给主会话
4) 立刻销毁子执行器的对话线程（不复用历史）
5) 主会话整合结果并做最终验收/打回

## 4. 并发与“写入冲突”处理
### 4.1 两类冲突
需要区分两类冲突：
1) 并发冲突：两个执行流同时写同一份工作区，导致覆盖/错乱（锁可以解决）。
2) 逻辑冲突：两个改动语义上不兼容（锁无法自动解决，只能检测 + 处理）。

### 4.2 推荐的最小可用策略：单写者 + patch 驱动
为了降低复杂度与风险，建议默认采用：
- 子任务可以并行做“读/分析/产出建议或 patch”
- 任何“真正落盘的写入动作”必须串行（单写者）
- 主会话作为最终写入者：
  - 先验收子任务输出（尤其是是否触及敏感文件、是否符合约束）
  - 再按顺序应用 patch / 执行写入命令

这相当于把“并发写文件”降维成“串行提交”，并让冲突变得可观测、可打回。

### 4.3 冲突出现后的处理策略
当发生 patch apply 失败或语义冲突时，主会话需要明确的打回策略：
- 重生成：把最新基线（最新代码片段/文件内容）发回给对应子任务，要求在新基线上重新生成 patch。
- 合并：由主会话（或专门的“合并子任务”）完成 merge，再执行验收命令。
- 放弃：若冲突成本过高或收益不大，直接取消该子任务的变更建议。

## 5. 记录“指派使用的模型名”
目标是让“这次任务到底用了哪个模型”可回溯、可统计、可验收。

建议记录策略：
- 将 `model` 作为任务卡的一部分落盘（例如存入本地任务记录）。
- 同时在日志/输出中打印 `model`，方便人工快速核对。

## 6. 与“项目隔离锁（per-project lock）”的关系
当前系统已经实现“跨项目并行、同项目串行”的执行边界（参考 ADR-0014）。
这个边界对本设计有两点意义：
- 正向：减少“不同入口同时改同一仓库”的风险（交互会话与任务队列互不打架）。
- 限制：一次请求内部若并发多个子任务，仍可能写冲突；因此本设计仍建议“写入串行”。

## 7. 风险与取舍
- 性能：用完即丢会增加冷启动次数；但换来上下文隔离与安全性。
- 成本：更快模型不一定更便宜；需要根据实际计费与效果调整策略。
- 可控性：若允许子执行器直接写文件，需要更严格的约束与验收，否则风险上升。

## 8. 开放问题（Deferred）
- 模型选择策略如何配置：
  - 是否引入一个配置文件，描述“任务类型 -> 使用模型”的规则。
  - 是否为不同模型定义擅长领域（frontend/backend/review）并由主会话据此指派。
- 子任务的权限模型：
  - 是否允许所有子任务写文件，还是默认只允许产出 patch/建议。
  - 是否按任务类型动态收缩工具权限。
