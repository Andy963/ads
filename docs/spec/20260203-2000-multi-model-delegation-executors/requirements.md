# 多模型指派（一次性子执行器）- 需求

## 1. 背景 / 问题
我们希望主对话（主 agent）保持长期稳定、上下文不丢失（例如固定使用一个“主模型”）。同时，针对一些简单/机械的工作，希望能使用更快的模型（例如一个“快模型”）来执行，从而降低等待时间与成本。

当前系统若通过“切换主 agent 的模型”来实现，会带来以下问题：
- 主对话上下文可能被重置（对话线程被重建），破坏“主会话永不受影响”的安全性目标。
- 如果尝试复用一个“池化的子执行器”，容易把上一次子任务的上下文带入下一次子任务，造成严重的上下文串扰。

因此需要一种机制：主会话固定且不被重置；每次指派时临时创建一个“子执行器”，指定模型、执行任务、返回结果后立刻销毁（不复用其对话历史）。

## 2. 目标（Goals）
- 主会话长期稳定：
  - 主 agent 的对话线程不因指派/换模型而被重置。
  - 主 agent 始终可继续接收用户输入并整合结果。
- 指派可指定模型（Model-as-Parameter）：
  - 每次指派可以明确指定使用的 `model`（如 `model-fast`）。
  - 指派使用与主会话相同的 `base_url` / `api_key` 配置来源，仅切换 `model`。
- 上下文隔离：
  - 子执行器默认“用完即丢”，不复用其对话历史（避免串上下文）。
  - 子执行器只接收本次任务需要的上下文输入（任务卡 + 必要文件路径/片段/约束）。
- 可观察性（Observability）：
  - 每次指派需要记录“实际使用的模型名”（用于验收、审计与回溯）。
  - 记录应尽量与“任务卡”绑定，而不是仅存在于日志里。
- 并发安全（最小可用版本）：
  - 在同一项目（workspace）内，写入工作区的动作需要串行（单写者），避免并发覆盖。
  - 允许跨项目并行（已有 per-project lock 的前提下）。

## 3. 非目标（Non-goals）
- 不追求自动解决复杂的语义冲突（例如两个改动逻辑互斥但都能 apply）。
- 不追求把“子执行器池化复用对话历史”作为默认（可作为未来优化，但默认禁用）。
- 不引入复杂的分布式调度/多进程 worker（本期只讨论单进程内的机制与接口）。

## 4. 关键约束（Constraints）
- 不把密钥写入 workspace：连接信息仍由现有 `.codex` 配置来源提供。
- 子执行器默认无状态（no history reuse），避免上下文泄漏。
- 同项目内默认串行写入；并发只用于“读/分析/产出 patch 或建议”。

## 5. 术语（Terminology）
- 主会话：用户与主 agent 的长期对话线程（稳定、可持续）。
- 指派 / 子任务：主会话把一部分工作交给后台执行器处理。
- 子执行器：一次性创建的后台执行器，使用指定 `model` 执行子任务，完成后销毁。
- 任务卡：描述子任务的结构化输入（目标、约束、交付物、验收方式、指定模型）。
- 单写者：同一时刻只有一个执行流可以修改工作区文件（写入串行）。

## 6. 验收与可观测性（Acceptance / Observability）
- 每次指派都能在输出/日志中看到：
  - `agentName`（可选）
  - `model`（必须）
  - 子任务摘要（goal + deliverables）
- 每次指派都能在本地状态存储中查询到：
  - 子任务 spec 中包含 `model` 字段（或等价字段），便于后续统计与审计。
