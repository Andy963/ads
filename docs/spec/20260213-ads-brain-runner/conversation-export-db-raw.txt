不会。前端“批准草稿变成正式任务”和“点击删除草稿”在后端是两条**不同的 API 路径/状态机**，只要你在做 spec 清理时以**后端状态变更**为准，就不会把批准当成删除。

**现状（代码层面已经能区分）**
- 批准：前端走 `POST /api/task-bundle-drafts/:id/approve`，后端把 draft 状态置为 `approved`（见 `src/web/server/api/routes/taskBundleDrafts.ts:137`；存储层 `approveTaskBundleDraft`）。
- 删除：前端走 `DELETE /api/task-bundle-drafts/:id`，后端把 draft 状态置为 `deleted`（见 `src/web/server/api/routes/taskBundleDrafts.ts:105`；存储层 `deleteTaskBundleDraft`）。

前端在批准后会调用 `deleteDraftLocal(rt, id)` 把草稿从面板里移除（见 `web/src/app/taskBundleDrafts.ts:105`），但这只是 UI 行为，不等于“删除”语义。

**你真正需要注意的坑**
- 当前 `DELETE /api/task-bundle-drafts/:id` 路由**没有检查 `existing.status`**（`approved` 也能被标记成 `deleted`，见 `src/web/server/api/routes/taskBundleDrafts.ts:105` 附近）。  
  如果你把“删除草稿 → 删除 spec”绑定到 `deleted`，那未来一旦有人误调用这个 DELETE，就可能把已批准任务对应的 spec 误删。

**推荐约束（用于实现 spec 清理时的判定）**
- 只有当 `existing.status === "draft"` 且本次请求是 `DELETE` 时，才允许进入“spec 清理/标记为可清理”的分支；否则返回 `409`。
- 批准路径（`approve`）永远不触发 spec 清理；它只会把状态置为 `approved`，spec 视为“保留”。

如果你确定要“用户删除草稿就删 spec”，我建议先做成“软删除/标记 orphan + 后台 GC”，避免误删手工改过或被复用的 spec 目录。你希望是硬删（直接 `fs.rm`）还是软删（移动到 `docs/spec/.trash/` 或打标等待 GC）？

- **注意力门控（attention gating）**：同一时刻只把极少量信号提升到“工作记忆”。大量输入被压到背景噪声，不进入当前推理回路。
- **工作记忆容量极小**：这反而是优势——它强迫“当前任务上下文”保持稀疏，减少交叉污染。
- **情景索引（contextual indexing）**：记忆不是按时间一锅端，而是按“线索/场景”索引（地点、人、目标、任务）。检索时由线索激活一簇相关内容，非相关内容不会被同时激活。
- **执行控制/抑制（executive control）**：会主动抑制无关联想，维持当前目标的稳定性。
- **多线程不共享同一“文本缓冲区”**：大脑不是把所有对话拼成一个长 prompt；它更像“很多索引 + 小片段按需取用”。

把这个类比回系统设计：`brain` 最像“执行控制 + 调度器”，而不是“万能对话模型本体”。

落到工程上，要避免串上下文/爆炸，关键是这几条约束：

- **强隔离的命名空间**：任何状态都挂在 `(workspaceId, channelId, conversationId)` 下；`worker` 只读自己的 namespace。
- **`brain` 只存指针，不存全文**：存 `jobId`、外部 `sessionId`、`prNumber`、状态机位置、最后检查时间、以及“可检索摘要/标签/哈希”；全文放对象存储/文件，按需加载。
- **事件驱动而非全文订阅**：`brain` 只看“事件/边沿变化”（例如 “Jules session completed”, “PR opened”, “Issue created”），而不是持续消费对话 token 流。
- **LLM 只在 bounded prompt 上工作**：每个 `worker` 都用 fresh `threadId`，prompt 由“代码快照 + 规则集 + 该 `jobId` 的少量历史摘要”拼出来，禁止注入其它项目的对话。
- **actor/queue 模型**：把每个 `workspaceId` 当成一个 actor；串台通常来自共享可变全局状态（同一个 thread、同一个 memory blob、同一个 scratchpad）。

你提到“人脑也是响应机制”，这点可以对应：`brain` 的 `tick`（定时检查）就是它的“外界刺激”。区别只在于：我们要显式规定 `tick` 只读取极小的状态与外部信号，然后把重活交给隔离的 `worker`，而不是让 `brain` 自己背着所有上下文推理。

不是“什么也没做”，而是 **brain 做的事不应该是“用 LLM 想一大段”**，而是做 **确定性的状态推进 + 调度**。它收到 `ingest report` 之后，至少会做这几件“硬动作”：

1) **路由与隔离**：按 `workspaceId` 把消息放进对应的 `mailbox/actor`（同一 workspace 串行处理，跨 workspace 可并行），避免串台。  
2) **幂等去重**：用 `eventId`/内容哈希/外部 ID（`julesSessionId/pr/issue`）做 dedupe，防止重复 ingest 导致重复任务。  
3) **更新状态机**：把 `facts/todos/external_refs` 写入该 workspace 的表（或带 `workspace_id` 的行），更新 `active_jobs`、`last_seen`、`next_check_at` 等。  
4) **生成后续调度**：把 `suggested_jobs[]` 变成真正的 `jobs` 入队（带 `run_at`、优先级、重试策略），例如：
   - “10min 后检查 `julesSessionId=...` 是否完成”
   - “发现 PR 已创建 → 触发 `pr_to_issue` job”
   - “发现 build 失败 → 触发 `triage_failure` job”
5) **触发执行**：如果有空闲 worker，就立即拉起/分配；否则排队等待。  
6) **可观测性/通知**：记录审计日志、指标；必要时把关键事件推送到 Telegram/Web（可选）。

你可以把 brain 视为“内核”：它不负责每次都深度推理，它负责把系统从**被动响应**变成**有状态的控制回路**——接收事件→推进状态→安排下一次检查/动作→驱动 worker 执行。

如果你觉得“这样 brain 还是太弱”，那通常是因为你把“理解事件”也算 brain 的职责。更稳的拆法是：**理解在 `ingest`（LLM）里做，决策落地在 brain（状态机）里做**。这就像 OS：用户态程序做复杂逻辑，内核只做调度与资源隔离。
