# 对话导出（节选）

说明：

- 这是对“runner/brain/ingest 架构讨论”的节选与整理版，不包含系统提示词、权限/规则大段重复文本。
- 目的：保留关键问题、关键结论、关键分歧点，便于后续继续推进设计与实现。

---

## 主题 1：为什么 Jules 更“客观”

- 你指出：直接让当前 agent 分析代码会带现有对话上下文，可能不客观；Jules 在独立空间分析，结果更客观。
- 我们确认：需要“上下文隔离 + 干净工作区”来接近 Jules 的效果。

## 主题 2：Telegram bot 的并发与后台任务冲突

- 你担心：TG 同时处理消息与后台任务会冲突，需要异步。
- 我们检查实现：TG 侧对同一用户是显式串行（如果有活跃请求，会拒绝新请求），并没有一个持久化后台调度器。

## 主题 3：常驻 runner 的价值

- 你希望：有一个后台常驻 runner，不受 web/telegram 重启影响，能跑长耗时/定时任务（代码审计、找 bug、提 issue）。
- 我们把它抽象为：需要一个常驻的 controller（brain）来做状态机推进与定时检查。

## 主题 4：入口与交互

- 你指出：纯 runner 如果只有执行面，没有入口就没法交互；除非通过当前 agent 委派任务。
- 我们收敛为：要拆出 control plane（web/tg 提交/查询/取消）与 data plane（runner/worker 执行），runner 不必依赖“对话式委派”。

## 主题 5：上下文爆炸与串台担忧

- 你强调：如果存在一个常驻 brain，它接收多个项目/多个角色的消息，很容易串上下文导致错乱。
- 我们收敛为：不要让 brain 成为“长期 LLM 对话 thread”；应当是“状态机 + 调度器”，LLM 只用于 bounded 的 ingest/worker 调用。

## 主题 6：自治 ingest 的必要性

- 你指出：如果不让 LLM 先读一遍消息，就无法判断是否无关；不能让人类先判断再喂给 LLM。
- 我们收敛为：每条消息都可以由一次性 `ingest` 调用来“读并提炼”，但这不意味着要把它持续追加进 brain 的上下文；提炼产物应结构化落库。

## 主题 7：可观测性（Web 可见）

- 你指出：如果 brain 直接 spawn 进程去做事，Web 用户看不到过程与结果，缺乏可观测性。
- 我们收敛为：brain 的动作应落到可观测实体（jobs/tasks/messages），最好复用现有 task queue 与 Web UI 展示能力。
