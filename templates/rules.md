# ADS 默认工作空间规则
## rules:
1. 数据库文件保护：未经明确许可，禁止删除或覆盖任何数据库文件。
2. 提交策略：除非用户明确要求，否则不得执行 `git commit`、`git push` 等提交操作，也不得添加 co-author。
3. 文档位置：所有新增文档必须位于 docs/ 目录内，严禁在其外部写文档。
4. 脚本创建：未经允许不得在项目中随意新建脚本或可执行文件。
5. 讨论阶段：方案讨论未结束或未得到批准时，禁止开始编写代码。
6. 前端构建：凡修改到前端相关代码（包含 src/frontend、前端依赖或 UI 组件），必须在交付前运行一次 `npm run build`，确保能通过编译。
7. 规划约束：多步骤任务必须先写出包含至少 2 步的计划，并在每个步骤完成后更新计划，禁止跳过或省略该流程。
8. 文档同步：任何改动导致行为、接口或流程发生变化时，必须同步更新对应的 `docs/spec/*`、README 或相关文档，保持实现与文档一致。
9. Spec 流程：除非用户明确要求跳过流程，否则新增或变更任何功能必须使用 `/ads.new` 创建工作流，并严格按 requirement → design → implementation 顺序定稿；禁止直接在 docs/spec 目录外手写文档或跳过步骤。
10. Review 流程：除非用户明确要求跳过且记录原因，实施完成后必须执行 `/ads.review` 并等待通过；Review 进行期间禁止继续修改代码或提交。
- 违反任一条即停止

## 基本规范

### 1. 保持工作区整洁
**规则**: 遵循已有的工程结构与代码风格，避免随意新增顶层目录或改变现有约定。

### 2. 任务驱动修改
**规则**: 仅在用户需求范围内修改文件；发现额外问题时先记录，再征求用户确认后处理。

## 禁止操作

### 1. 禁止泄露敏感信息
**规则**: 不得在代码、日志或输出中写入密钥、口令、令牌等任何敏感凭据。

### 2. 禁止破坏性删除
**规则**: 未获确认不得删除数据库、二进制产物或 Workspace 关键文件 (.ads 目录、docs/spec 等)。

### 3. 禁止擅自执行危险命令
**规则**: 不执行未经用户许可的网络、权限提升或依赖安装命令。

## 交付要求

### 1. 严格测试验证
**规则**: 在宣告任务完成前运行项目要求的测试、构建或检查流程，确保全部通过。

### 2. 清晰沟通与记录
**规则**: 关键决策、权衡与未解决事项需要在回复中说明，并指向相关文件或步骤。
